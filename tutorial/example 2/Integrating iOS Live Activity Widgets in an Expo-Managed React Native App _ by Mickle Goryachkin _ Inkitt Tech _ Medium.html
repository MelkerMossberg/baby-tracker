Integrating iOS Live Activity Widgets in an Expo-Managed React Native App

Mickle Goryachkin
9 min read
·
May 14, 2025

Hi all! I am Mikhail, a software engineer at Inkitt, and I want to share my adventure of the creating a widget for one of our RN projects. I hope this can help others who might have similar questions about this implementation. I have previous experience working with creating native widgets for React Native applications. One widget I built displayed a countdown timer to unlock the next chapter, and it looked like this:

The experiment was successful and the product team decided to implement the same functionality to another application. But here’s where I I faced my first roadblock — this new project this is a full Expo application, and as we know, Expo has some unique challenges when it comes to integrating native functionality.

Problem: Registering a New Target

Since the widget is a new target, we need to register it in our Xcode project. However there is a catch: every time we deploy the app, the iOS folder will be recreated, which means any manual change made in Xcode would be lost.

Solution: Apple Targets Plugin

Let’s start

At first, I thought that since the same problem exists for native components at the Expo, I assumed that there was a similar solution for creating new targets, and after some searching, I found only one working solution — Apple Targets plugin.

This plugin allows you to register new targets for your application outside of the iOS folder, and even links them to your project and all your extension files will be available inside the virtual folder and you can edit them directly from Xcode. Magic, right? Damn it, it is.
Before getting started, we need to make sure the environment meets the minimum requirements:

  • CocoaPods 1.16.2 (ruby 3.2.0)
  • Xcode 16 (macOS 15 Sequoia)
  • Expo SDK 52

Once everything is ready, let’s move on to the setup.

Creating a New iOS Target

Assuming that you are using a fresh starter Expo project (blank Expo application), the first thing we need to do is add @bacons/apple-targets as a dependency in your package.json. Then add this plugin to our expo project configuration file (app.json or app.config.*):

"plugins": [
  "@bacons/apple-targets"
]

Next, we’ll create a new target for our expo App:

npx create-target

If is all ok — we will see the menu with the what type of widget we want to create:

Very first option is likely what we need. After selecting it, just follow the further installation instructions. By default this plugin works with the app.config.* file and if you’re using app.json (as I did), you will see a message stating that the script cannot automatically update the configuration file, so you will need to do it manually. Not a problem for us, we just need to add these to your configuration:

"ios": {
   "appleTeamId": "your app team id",
   "infoPlist": {
     "NSSupportsLiveActivities": true
   },
   "entitlements": {
     "com.apple.security.application-groups": ["group.your-app-domain"]
   }
}

We’ve configured our app to use the Live Activity widget and set up a shared group between the main app and widget. This allows them to share data. . If all was set up correctly (I don’t doubt it 😅) — we will see new catalog in the root project folder: /targets/widget. The default set files in new directory include:

  • AppIntent.swift
  • Info.plist
  • WidgetControl.swift
  • WidgetLiveActivity.swift
  • expo-target.config.js
  • index.swift
  • widgets.swift

These files support for all types of widgets available in iOS. However, since we only need the Live Activity widget, we can delete the unnecessary files. Let’s do it:

rm -rf targets/widget/WidgetControl.swift targets/widget/widgets.swift targets/widget/AppIntent.swift

After this, open the file index.swift. Here you can see the initialization for the all widget types. Remove the inititialization code for the other widget types, leaving only WidgetLiveActivity. In the final you should get this:

import WidgetKit
import SwiftUI

@main
struct exportWidgets: WidgetBundle {
    var body: some Widget {
        // Export widgets here
        WidgetLiveActivity()
    }
}

We are very close. Add one more change in the file expo-target.config.js. frameworks for needed for the UI native layer (SwiftUI, ActivityKit):

/** @type {import('@bacons/apple-targets/app.plugin').ConfigFunction} */
module.exports = (config) => ({
  type: 'widget',
  // icon: 'https://github.com/expo.png',
  entitlements: {
    /* Add entitlements */
  },
  frameworks: ['SwiftUI', 'ActivityKit'],
});

Finally, we can try to run our app:

rm -rf ios
npx expo prebuild -p ios
npx expo run:ios

Once that’s done, check the the Podfile to confirm that the widget is dynamically loaded:

# apple-targets-extension-loader -- Dynamic loading of target configurations
Dir.glob(File.join(__dir__, '..', 'targets', '**', 'pods.rb')).each do |target_file|
  target_name = File.basename(File.dirname(target_file))
  target target_name do
    # Create a new binding with access to necessary methods and variables
    target_binding = binding
    target_binding.local_variable_set(:podfile_properties, podfile_properties)

    # Evaluate the target file content in the new binding
    eval(File.read(target_file), target_binding, target_file)
  end
end

This should be enough for your application to automatically register your widget every time for each build of the application. Now, when you open Xcode, you’ll see a link to our widget folder, and you can make any changes to the widget directly in Xcode. This is really awesome!

Understanding the widget Code

Here is an example from my project — you will see some default files. For us, the file of interest is: WidgetLiveActivity.swift.

import ActivityKit
import WidgetKit
import SwiftUI

struct WidgetAttributes: ActivityAttributes {
    public struct ContentState: Codable, Hashable {
        // Dynamic stateful properties about your activity go here!
        var emoji: String
    }

    // Fixed non-changing properties about your activity go here!
    var name: String
}

struct WidgetLiveActivity: Widget {
    var body: some WidgetConfiguration {
        ActivityConfiguration(for: WidgetAttributes.self) { context in
            // Lock screen/banner UI goes here
            VStack {
                Text("Hello \(context.state.emoji)")
            }
            .activityBackgroundTint(Color.cyan)
            .activitySystemActionForegroundColor(Color.black)

        } dynamicIsland: { context in
            DynamicIsland {
                // Expanded UI goes here.  Compose the expanded UI through
                // various regions, like leading/trailing/center/bottom
                DynamicIslandExpandedRegion(.leading) {
                    Text("Leading")
                }
                DynamicIslandExpandedRegion(.trailing) {
                    Text("Trailing")
                }
                DynamicIslandExpandedRegion(.bottom) {
                    Text("Bottom \(context.state.emoji)")
                    // more content
                }
            } compactLeading: {
                Text("L")
            } compactTrailing: {
                Text("T \(context.state.emoji)")
            } minimal: {
                Text(context.state.emoji)
            }
            .widgetURL(URL(string: "https://www.expo.dev"))
            .keylineTint(Color.red)
        }
    }
}

extension WidgetAttributes {
    fileprivate static var preview: WidgetAttributes {
        WidgetAttributes(name: "World")
    }
}

extension WidgetAttributes.ContentState {
    fileprivate static var smiley: WidgetAttributes.ContentState {
        WidgetAttributes.ContentState(emoji: "😀")
     }
     
     fileprivate static var starEyes: WidgetAttributes.ContentState {
         WidgetAttributes.ContentState(emoji: "🤩")
     }
}

#Preview("Notification", as: .content, using: WidgetAttributes.preview) {
   WidgetLiveActivity()
} contentStates: {
    WidgetAttributes.ContentState.smiley
    WidgetAttributes.ContentState.starEyes
}

In the WidgetActivity.swift file, you’ll see the basic structure for the live activity widget. The example is a widget that displays a dynamic emoji.You can adjust the UI for more advanced features for Live Activities. You can find a full description of which part is responsible for which display here:
https://developer.apple.com/documentation/activitykit/displaying-live-data-with-live-activities.

Controlling the widget from the App

Now we need to add minimal functionality to start and stop our widget from the app. First, we’ll need to create a simple native module that sends commands to the widget.

Here we have a very good documentation for what we need to do in order to create a native module for our Expo app — https://docs.expo.dev/modules/native-module-tutorial/. Let’s do it:

npx create-expo-module expo-live-controller --local

You should see something like this:

This command generate new folder, with your module name, inside modules folder in your app:

Here we see that the script generated a cross platform module structure, however we only need the iOS-specific module code.. In this case we need to do a little cleanup. We need to remove these files and folders. In result we should have this structure:

After this we need to open the file: expo-module.config.json and save configuration only for IOS.

{
  "platforms": [
    "apple"
  ],
  "apple": {
    "modules": [
      "ExpoLiveControllerModule"
    ]
  }
}

Now let’s create two simple methods for our widget — start (start Live Activity action) and stop. For this we need to open the ExpoLiveControllerModule.ts and add this code in this file:

import { requireNativeModule } from 'expo';
import { Platform } from 'react-native';

const nativeModule = Platform.OS === 'android' ? null : requireNativeModule('ExpoLiveController');

export const LiveActivityNativeModule = nativeModule
  ? nativeModule
  : {
      start: (data: string) => ({
        isSuccess: false,
      }),
      stop: (id: string) => ({
        isSuccess: false,
      }),
    };

Here we register the native module in our app for the IOS devices. For Android, we just create placeholder methods. Since we are not anarchists 😇, for the sake of order, let’s add a few types to describe these methods in the ExpoLiveController.types.ts

type BaseActivityResponse = {
  isSuccess: boolean;
  message?: string;
};

export type ActivityStartResponse = {
  id?: string;
} & BaseActivityResponse;

export type ActivityStopResponse = {
  id?: string;
} & BaseActivityResponse;

export type StartLiveActivityFn = (data: string) => Promise<ActivityStartResponse>;

export type StopLiveActivityFn = (id: string) => Promise<ActivityStopResponse>;

Nice, we did a good job! Now all we need to do is change the file ExpoLiveControllerModule.swift so that our JS code can communicate with the native part of the application.

import ActivityKit
import SwiftUI
import ExpoModulesCore

final class UnavailableException: GenericException<Void> {
    override var reason: String {
        "Live activities are not available on this system."
    }
}

struct WidgetAttributes: ActivityAttributes {
    public struct ContentState: Codable, Hashable {
        // Dynamic stateful properties about your activity go here!
        var emoji: String
    }

    // Fixed non-changing properties about your activity go here!
    var name: String
}

struct StartActivityReturnType: Record {
    @Field
    var id: String?
    
    @Field
    var message: String?
    
    @Field
    var isSuccess: Bool
}

struct StopActivityReturnType: Record {
    @Field
    var id: String?
    
    @Field
    var message: String?
    
    @Field
    var isSuccess: Bool
}

// MARK: Module definition

public class ExpoLiveControllerModule: Module {
    public func definition() -> ModuleDefinition {
        Name("ExpoLiveControllerModule")
        
        AsyncFunction("start") {
            (
                data: String,
                promise: Promise
            ) in
            guard #available(iOS 16.2, *) else {
                throw UnavailableException(())
            }
            
            let info = ActivityAuthorizationInfo()
            guard info.areActivitiesEnabled else {
                throw UnavailableException(())
            }
            
            let initialContentState = WidgetAttributes.ContentState(emoji: data)
            let activityAttributes = WidgetAttributes()
            
            let activityContent = ActivityContent(state: initialContentState)
            
            do {
                let activity = try Activity.request(attributes: activityAttributes, content: activityContent)
                if let id = activity.id as String? {
                    return promise.resolve(StartActivityReturnType(id: Field(wrappedValue: id), isSuccess: Field(wrappedValue: true)))
                } else {
                    return promise.resolve(StartActivityReturnType(message: Field(wrappedValue: "Failed to create an activity based on params: \(initialContentState.emoji)"), isSuccess: Field(wrappedValue: false)))
                }
            } catch {
                return promise.resolve(StartActivityReturnType(message: Field(wrappedValue: "Error requesting Live Activity \(error.localizedDescription)."), isSuccess: Field(wrappedValue: false)))
            }
        }
        
        AsyncFunction("stop") { (id: String, promise: Promise) in
            guard #available(iOS 16.2, *) else {
                throw UnavailableException(())
            }
            
            log.debug("Stopping activity \(id)")
            
            Task {
                if let activity = Activity<WidgetAttributes>.activities.first(where: { $0.id == id }) {
                    await activity.end(dismissalPolicy: .immediate)
                    return promise.resolve(StopActivityReturnType(id: Field(wrappedValue: id), isSuccess: Field(wrappedValue: true)))
                } else {
                    return promise.resolve(StopActivityReturnType(message: Field(wrappedValue: "Activity with ID \(id) not found"), isSuccess: Field(wrappedValue: false)))
                }
            }
        }
    }
}

Now we can run the app and use this code:

LiveActivityNativeModule.start("🥰")

for the stop activity:

LiveActivityNativeModule.stop("here need to use activity id what return start method")

After this on the your device lock screen you will see the our widget:

This post covered the basics of integrating iOS Live Activity widget into an Expo-managed React Native app. In future posts, I’ll dive into displaying images, creating countdown timers, and automating deployment with Fastlane. So stay tuned.
